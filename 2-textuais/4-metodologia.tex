\section{Definição dos requisitos}
O primeiro passo no desenvolvimento de \textit{software} é a definição dos requisitos, com eles tem-se um direcionamento sobre o que deve ser feito. A partir da revisão bibliográfica foram definidos alguns requisitos básicos que deveriam estar presentes na ferramenta. Apesar das ferramentas compartilharem as mesmas funcionalidades básicas já definidas inicialmente, outras têm características interessantes que podem ser reaproveitadas. Como requisito não funcional foi definido oferecer suporte multi-plataforma, para \textit{mobile}, \textit{desktop} e \textit{web}. Como requisitos funcionais foram definidos os seguintes:

\begin{itemize}[label={$\sbullet$}]
  \item Permitir que os usuários digitem uma gramática para ser analisada.
  \item Permitir que os usuários visualizem o estado das estruturas dos algoritmos.
  \item Permitir que os usuários avancem, retornem e reiniciem os passos da execução dos algoritmos.
  \item Permitir que os usuários selecionem o algoritmo a ser visualizado.
  \item Permitir que os usuários acessem o pseudocódigo dos algoritmos.
  \item Permitir que os usuários adicionem \textit{breakpoints} no pseudocódigo.
  \item Permitir que os usuários visualizem o autômato dos algoritmos \gls{lr}.
  \item Permitir que os usuários visualizem uma descrição dos botões da interface.
  \item Permitir que os usuários digitem uma \textit{string} para ser analisada.
  \item Permitir que os usuários visualizem a análise de uma \textit{string}.
  \item Permitir que os usuários visualizem a árvore sintática de uma \textit{string}.
  \item Permitir que os usuários copiem em formato de texto o resultado dos algoritmos.
\end{itemize}

\section{Definição da arquitetura do projeto}
O projeto será construído usando o \textit{framework Svelte}, sem \gls{ssr}, para ser possível o funcionamento \textit{offline} da ferramenta, já que a ferramenta não teria acesso ao servidor não seria possível usá-lo para renderizar elementos. \textit{Svelte} compila a base de código e cria uma coleção de arquivos estáticos que constituem a página \textit{web} e a base para o suporte multi-plataforma da aplicação.

\begin{mypicture}{\label{fig:arqremo}}{Arquitetura da aplicação em nuvem}{figuras/remote.png}{fornecida pelo próprio autor}
\end{mypicture}

O \textit{framework Capacitor}\footnote{https://capacitorjs.com} consome essa coleção de arquivos e cria um projeto para plataforma \textit{Android} usado para criar a versão \textit{mobile} da ferramenta usando o \textit{Android Studio}. O \textit{framework Tauri}\footnote{https://tauri.app} constrói instaladores para \textit{desktop} diretamente da coleção de arquivos estáticos. A plataforma alvo dos instaladores é a plataforma em que eles são construídos, já que o \textit{framework} não tem suporte para construção \textit{cross-platform} é necessária a utilização de máquinas virtuais para construir instaladores para diferentes plataformas \textit{desktop}.

Para a versão \textit{online} da aplicação, os serviços  de computação em nuvem \textit{Netlify}\footnote{https://www.netlify.com} e \textit{Supabase}\footnote{https://supabase.com} serão usados para hospedar respectivamente a aplicação e a data-base usada por ela. \textit{Netlify} e \textit{Supabase} foram escolhidos pela facilidade de uso e pelo oferecimento gratuito dos serviços para aplicações pequenas como a proposta nesse trabalho.

O projeto da aplicação foi hospedado em um repositório no \textit{Github}\footnote{https://github.com/fagner02/parsers-svelte.git} que é uma plataforma \textit{web} de hospedagem de repositórios do sistema de versionamento \textit{Git}\footnote{https://git-scm.com}. Graças a integração entre \textit{Netlify} e \textit{Github}, a aplicação é atualizada automaticamente com cada \textit{commit} feito para o repositório do projeto. Por fim, a plataforma \textit{Supabase} será utilizada para armazenar arquivos de \textit{log} da utilização da aplicação durante os testes.

\section{Integração dos algoritmos à ferramenta}
O primeiro passo da integração dos algoritmos é a criação de elementos gráficos representando as estruturas de dados usados nesses algoritmos. Foram criados em total seis elementos para as estruturas de pilha, conjunto, tabela(matriz), estado de autômato, autômato e árvore sintática.

Os quatro primeiros elementos são caixas criadas com \gls{html} já que seus dados podem ser representados com simples listas de texto. Na parte inferior dessas caixas está o título do elemento e na parte superior estão elementos textuais mostrando os dados das estruturas com destaques de cor nas chaves dos conjuntos e tabelas. Os dois últimos elementos são criados com \gls{svg} já que seus dados precisam ser estruturados de forma complexas.

Com os elementos criados para representar as estruturas é possível mostrar os passos da execução atualizando os dados desses elementos conforme as mudanças feitas nas estruturas. Após a sincronização dos elementos com os passos de execução é possível criar o controle de fluxo.

A primeira forma de implementação do controle de fluxo usada na aplicação foi usar uma única função que modifica as estruturas de dados e atualiza os elementos da interface em simultâneo. Essa função é controlada usando os mecanismos de execução assíncrona do \textit{JavasScript} sendo eles \textit{async}, \textit{await} e \textit{Promises}.

Nessa implementação as pausas são feitas usando uma \textit{Promise} com retorno indefinido. A ação de ir para o próximo passo é feita rejeitando a \textit{Promise} o que resume a execução da função. E por fim a ação de voltar, pular e reiniciar são feitas retornando uma nova chamada da função e executando a função e pulando as pausas até chegar no passo esperado.

O problema com a forma de implementação citada acima é que os passos da execução não são pré-calculados. Por essa razão, se a execução de um algoritmo tiver uma quantidade muito grande de passos, quando a ação de voltar ou pular for performada, vários passos serão executados em sequência até chegar no passo esperado.

Dessa forma, dado o fato que funções de atualização da interface também serão executadas, a aplicação vai requerer uma quantidade maior de poder computacional e vai parar de responder até que a execução dos passos seja finalizada.

Pela razão citada acima foi necessário criar uma segunda forma de implementação de controle de fluxo. Nessa implementação o estado das estruturas em cada passo é salvo em uma \textit{array} usando apenas as estruturas de dados sem modificar a interface. Dessa forma, a aplicação consegue pré-calcular os passos da execução sem muito esforço. Assim, quando as ações de controle de fluxo são performadas, os estados salvos são carregados e a interface é atualizada de acordo com eles.

\section{Implementação das animações dos algoritmos}
As mudanças de estados que ocorrem nos algoritmos podem ser melhor compreendidas se poderem ser visualizadas como transições ao invés de mostrar as mudanças saltando do estado inicial para o estado final. Usar animações torna a visualização das mudanças muito mais dinâmica.

As primeiras animações implementadas são as transições de estilo usando a propriedade \textit{CSS transition} que aplica uma transição suave entre mudanças de estilo dos elementos como tamanho e cor. As animações seguintes foram implementadas usando \textit{JavasScript} para serem atualizadas dinamicamente a medida que os elementos fossem modificados mudando de tamanho ou posição na interface.

Começando com a animação de destaque dos dados dentro de um elemento da interface. Essa animação é usada quando um dado de um elemento é utilizado, seja quando um item é adicionado a um elemento ou quando um item é usado como chave para acessar o item de outro elemento. Essa animação põe um item em destaque com um bloco colorido exibido atrás do item. Quando itens são adicionados aos elementos de tabela, as linhas e colunas também são destacadas com uma cor de fundo.

Vários elementos têm iterações feitas sobre seus itens dispostos na vertical. Para demonstrar cada iteração, uma moldura se movimenta sobre os elementos deixando claro a qual item cada iteração se refere. Alguns elementos têm itens que são listas de itens dispostos na horizontal. Para demonstrar iterações sobre essas listas de itens a animação de destaque citada no parágrafo anterior é usada para cada iteração.

Alguns dados são movidos entre elementos como no algoritmo \textit{first} que move símbolos da direita de uma produção para o conjunto \textit{first}. Para demonstrar essa transferência entre os elementos uma linha é exibida que se estende do item de origem até o elemento de destino com uma animação.

\section{Avaliação da Ferramenta}
\label{sec:avaliacao}

A avaliação da ferramenta VANSI foi estruturada para investigar tanto sua usabilidade quanto sua eficácia pedagógica. O estudo adotou uma abordagem metodológica mista, combinando coleta automatizada de dados de interação com questionários estruturados de \textit{feedback}.

\subsection{Participantes}

O estudo envolveu 23 estudantes voluntários que haviam cursado a disciplina de Compiladores do curso de Ciência da Computação do Campus Quixadá da Universidade Federal do Ceará.

\subsection{Instrumentos de Coleta de Dados}

Para garantir uma coleta abrangente e minimizar vieses, foram empregados três instrumentos complementares, cada um focado em diferentes dimensões da experiência do usuário. O primeiro instrumento consistiu no sistema de \textit{log} integrado à própria ferramenta VANSI. Este sistema registra automaticamente, em intervalos de 500 milissegundos, diversos eventos de interação do usuário com a interface.

Entre os dados capturados destacam-se as coordenadas do cursor do mouse, eventos de clique, transições entre as diferentes abas da interface (Gramática, LL(1), SLR, LR(1), Análise de \textit{String}), interações com elementos dinâmicos como \textit{tooltips} e janelas flutuantes, ações sobre os controles de fluxo de execução, tempos de permanência em cada componente e valores inseridos em campos de formulário. A estrutura de armazenamento desses dados segue o esquema apresentado na Tabela \ref{qua:esquema-log}, que organiza os registros em formato CSV para posterior processamento.

\begin{myquadro}{\label{qua:esquema-log}}{Esquema dos dados registrados pelo sistema de \textit{log} da VANSI}{llX}[hvlines]

  \textbf{Campo}     & \textbf{Tipo} & \textbf{Descrição}                                         \\

  timestamp          & DateTime      & Data e hora do evento com precisão de milissegundos        \\
  type               & String        & Categoria do evento (mouse pos, w click, tab change, etc.) \\
  desc               & String        & Descrição contextual e valores associados ao evento        \\
  active\_tab        & String        & Identificador da aba ativa no momento da interação         \\
  mouse\_x, mouse\_y & Integer       & Coordenadas do cursor na interface gráfica                 \\
  window\_id         & String        & Identificador da janela flutuante envolvida na interação   \\
\end{myquadro}

O segundo instrumento de coleta foi uma atividade prática composta por oito tarefas progressivas, elaboradas para exercitar as principais funcionalidades da ferramenta e avaliar sua completude. A sequência de tarefas iniciou com o cálculo dos conjuntos \textit{First} e \textit{Follow} para uma gramática fornecida, avançou para a identificação e correção de conflitos \gls{ll}(1), e prosseguiu com a construção completa da tabela de \textit{parsing} \gls{ll}(1). Em seguida, os participantes foram direcionados para a geração do autômato \gls{lr}(0) para uma gramática \gls{slr}, seguida da construção da respectiva tabela de ações. A sexta tarefa envolveu a construção do autômato \gls{lr}(1) com itens canônicos, enquanto as duas últimas focaram na aplicação prática, com a análise sintática de \textit{strings} utilizando as tabelas \gls{ll}(1) e \gls{slr} previamente construídas. Este conjunto de tarefas foi projetado para verificar a funcionalidade técnica da ferramenta e para avaliar sua efetividade como recurso pedagógico.

Complementando os dados objetivos de interação, o terceiro instrumento de coleta consistiu em um questionário estruturado de \textit{feedback} aplicado imediatamente após a conclusão da atividade prática. O questionário, composto por 21 questões, foi organizado em quatro seções principais, conforme detalhado na Tabela \ref{qua:questionario-estrutura}. A seção de informações gerais buscou caracterizar o perfil dos participantes em relação à experiência prévia e nível de conhecimento. A seção de interface e usabilidade avaliou aspectos como facilidade de navegação, clareza dos elementos e organização do \textit{layout}. A seção de recursos pedagógicos trata da utilidade percebida das diferentes visualizações, animações e mecanismos de controle. Por fim, a avaliação geral coletou impressões holísticas sobre a ferramenta e intenções de uso futuro. A construção do instrumento combinou escalas \textit{Likert} de cinco pontos para questões quantitativas com alternativas fechadas para questões categóricas, reservando uma questão aberta final para comentários e sugestões livres.

\begin{myquadro}{\label{qua:questionario-estrutura}}{Estrutura de seções do questionário de feedback}{llX}[hvlines]
  \textbf{Seção}       & \textbf{Nº Questões} & \textbf{Objetivo Específico}                                              \\

  Informações Gerais      & 3                    & Caracterização do perfil e experiência prévia dos participantes           \\
  Interface e Usabilidade & 7                    & Avaliação da navegabilidade, clareza visual e organização dos elementos   \\
  Recursos Pedagógicos    & 7                    & Análise da efetividade das visualizações, animações e controles didáticos \\
  Avaliação Geral         & 4                    & Coleta de impressões holísticas e intenções de uso futuro                 \\

\subsection{Metodologia de Análise}

A análise dos dados coletados adotou uma abordagem de métodos mistos, integrando técnicas quantitativas e qualitativas para uma compreensão multidimensional dos resultados. A análise quantitativa concentrou-se inicialmente na estatística descritiva das métricas de uso extraídas dos \textit{logs}, calculando médias, medianas, desvios padrão e distribuições de frequência para os diferentes tipos de interação.

Além disso, técnicas de visualização foram empregadas com a geração de \textit{heatmaps} de distribuição de cliques e linhas do tempo de interação, facilitando a identificação de padrões comportamentais. A identificação desses padrões foi feita manualmente através da comparação entre as vizualizações geradas e o \textit{feedback} dos participantes.

Os \textit{heatmaps} de cliques foram gerados utilizando as coordenadas de clique registradas nos \textit{logs}, aplicando um mapa de calor sobre uma imagem interface da ferramenta para destacar as áreas de maior interação. As imagens da interface presentes nos \textit{heatmaps} foram geradas automaticamente por um \textit{script JavaScript} executado no \textit{Node.js}. O \textit{script} utiliza a biblioteca \textit{puppeteer} para lançar uma instância de um navegador, acessar o \textit{url} da ferramenta, navegar até a tela desejada e fazer uma captura de tela tudo de forma automatizada. O \text{script} está disponível publicamente no repositório do projeto\footnote{https://github.com/fagner02/parsers-svelte.git}.

As linhas do tempo de interação foram construídas plotando os eventos registrados ao longo do tempo, permitindo a visualização de sequências e durações de uso dos diferentes componentes da interface. Os gráficos de linha de tempo também marcam as transições entre as abas que são identificadas por cliques em elementos de navegação registrados nos \textit{logs}.

Por fim, a análise qualitativa focou nos comentários livres do questionário. Essa análise buscou identificar temas recorrentes, sugestões de melhoria e percepções subjetivas que complementassem os dados quantitativos. A análise feita a partir de multiplas fontes, integrando dados de \textit{logs}, respostas escalares e comentários textuais, permitiu uma validação cruzada dos achados e uma compreensão mais rica da experiência dos usuários.

\subsection{Ferramentas e Considerações Técnicas}

Para o processamento e análise dos dados, foram desenvolvidos \textit{scripts} em \textit{Python} que utilizam as bibliotecas \textit{pandas} e \textit{numpy} para manipulação e análise estatística dos dados tabulares, \textit{matplotlib} e \textit{seaborn} para geração de visualizações e gráficos, e \textit{scipy} para cálculo de correlações estatísticas e testes de hipóteses. Todo o código desenvolvido para a análise está disponível publicamente no repositório do projeto\footnote{https://github.com/fagner02/parsers-svelte.git}, garantindo a transparência e reprodutibilidade dos procedimentos analíticos.

A condução do estudo também considerou aspectos éticos da pesquisa envolvendo seres humanos. Foi obtido o consentimento informado de todos os participantes, garantindo o anonimato completo dos dados pessoais e a utilização exclusiva das informações para fins acadêmicos e de pesquisa.

Esta metodologia de avaliação, ao integrar dados objetivos de interação com percepções subjetivas dos usuários, permitirá não apenas quantificar a aceitação da ferramenta, mas também compreender os mecanismos pelos quais a visualização interativa pode facilitar o aprendizado de algoritmos complexos de análise sintática. Os resultados dessa análise serão apresentados e discutidos no capítulo seguinte, fornecendo evidências empíricas sobre a efetividade da VANSI como recurso educacional.
