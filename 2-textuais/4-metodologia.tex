\section{Definição dos requisitos}
O primeiro passo no desenvolvimento de \textit{software} é a definição dos requisitos. Com eles tem-se um direcionamento sobre o que deve ser feito. A partir da revisão bibliográfica foram definidos alguns requisitos básicos que deveriam estar presentes na ferramenta. Apesar das ferramentas compartilharem as mesmas funcionalidades básicas já definidas inicialmente, outras têm características interessantes que podem ser reaproveitadas. Como requisito não funcional foi definido oferecer suporte multi-plataforma, para \textit{mobile}, \textit{desktop} e \textit{web}. Como requisitos funcionais foram definidos os seguintes:

\begin{itemize}[label={$\sbullet$}]
    \item Permitir que os usuários digitem uma gramática para ser analisada.
    \item Permitir que os usuários visualizem o estado das estruturas dos algoritmos.
    \item Permitir que os usuários avancem, retornem e reiniciem os passos da execução dos algoritmos.
    \item Permitir que os usuários selecionem o algoritmo a ser visualizado.
    \item Permitir que os usuários acessem o pseudocódigo dos algoritmos.
    \item Permitir que os usuários adicionem \textit{breakpoints} no pseudocódigo.
    \item Permitir que os usuários visualizem o autômato dos algoritmos \gls{lr}.
    \item Permitir que os usuários visualizem uma descrição dos botões da interface.
    \item Permitir que os usuários digitem uma \textit{string} para ser analisada.
    \item Permitir que os usuários visualizem a análise de uma \textit{string}.
    \item Permitir que os usuários visualizem a árvore sintática de uma \textit{string}.
    \item Permitir que os usuários copiem em formato de texto o resultado dos algoritmos.
\end{itemize}

\section{Definição da arquitetura do projeto}
O projeto será construído usando o \textit{framework Svelte}, sem \gls{ssr}, para ser possível o funcionamento \textit{offline} da ferramenta. Já que a ferramenta não teria acesso ao servidor quando \textit{offline}, não seria possível usá-lo para renderizar elementos. \textit{Svelte} compila a base de código e cria uma coleção de arquivos estáticos que constituem a página \textit{web} e a base para o suporte multi-plataforma da aplicação.

\begin{mypicture}{\label{fig:arqremo}}{Arquitetura da aplicação em nuvem}{figuras/remote.png}{fornecida pelo próprio autor}
\end{mypicture}

O \textit{framework Capacitor}\footnote{https://capacitorjs.com} consome essa coleção de arquivos e cria um projeto para a plataforma \textit{Android}. Esse projeto é usado para criar a versão \textit{mobile} da ferramenta usando o \textit{Android Studio}. O \textit{framework Tauri}\footnote{https://tauri.app} constrói instaladores para \textit{desktop} diretamente da coleção de arquivos estáticos. A plataforma alvo dos instaladores é a plataforma em que eles são construídos. Já que o \textit{framework} não tem suporte para construção \textit{cross-platform} é necessária a utilização de máquinas virtuais para construir instaladores para diferentes plataformas \textit{desktop}.

Para a versão \textit{online} da aplicação, os serviços  de computação em nuvem \textit{Netlify}\footnote{https://www.netlify.com} e \textit{Supabase}\footnote{https://supabase.com} serão usados para hospedar respectivamente a aplicação e a data-base usada por ela. \textit{Netlify} e \textit{Supabase} foram escolhidos pela facilidade de uso e pelo oferecimento gratuito dos serviços para aplicações pequenas como a proposta neste trabalho.

O projeto da aplicação foi hospedado em um repositório\footnote{https://github.com/fagner02/parsers-svelte.git} no \textit{Github} que é uma plataforma \textit{web} de hospedagem de repositórios do sistema de versionamento \textit{Git}\footnote{https://git-scm.com}. Graças a integração entre \textit{Netlify} e \textit{Github}, a aplicação é atualizada automaticamente com cada \textit{commit} feito para o repositório do projeto. Por fim, a plataforma \textit{Supabase} será utilizada para armazenar arquivos de \textit{log} da utilização da aplicação durante os testes.

\section{Integração dos algoritmos à ferramenta}
O primeiro passo da integração dos algoritmos é a criação de elementos gráficos representando as estruturas de dados usadas nesses algoritmos. Foram criados, no total, seis elementos para as estruturas de pilha, conjunto, tabela (ou matriz), autômato, estado de autômato e árvore sintática.

Os quatro primeiros elementos dos seis citados acima são caixas criadas com \gls{html}, já que seus dados podem ser representados com simples listas de texto. Na parte inferior dessas caixas está o título do elemento e, na parte superior, estão elementos textuais, mostrando os dados das estruturas com destaques de cor nas chaves dos conjuntos e tabelas. Os dois últimos elementos são criados com \gls{svg}, já que seus dados precisam ser estruturados de forma complexa.

Com os elementos criados para representar as estruturas, é possível mostrar os passos da execução atualizando os dados desses elementos, conforme as mudanças feitas nas estruturas. Após a sincronização dos elementos com os passos de execução, é possível criar o controle de fluxo.

A primeira forma de implementação do controle de fluxo usada na aplicação foi usar uma única função que modifica as estruturas de dados e atualiza os elementos da interface em simultâneo. Essa função era controlada usando os mecanismos de execução assíncrona do \textit{JavaScript} sendo eles \textit{async}, \textit{await} e \textit{Promises}.

Nessa implementação, as pausas são feitas chamando uma \textit{Promise} com retorno \textit{undefined}, interrompendo a execução da função. A ação de ir para o próximo passo é feita rejeitando a \textit{Promise}, o que retoma a execução da função. E por fim, as ações de voltar, pular e reiniciar são feitas retornando uma nova chamada da função. A nova chamada é executada e as pausas são puladas (as \textit{Promises} são rejeitadas) até chegar no passo esperado.

O problema com a forma de implementação citada acima é que os passos da execução não são pré-calculados. Por essa razão, se a execução de um algoritmo tiver uma quantidade muito grande de passos, quando a ação de voltar ou pular for realizada, vários passos serão executados em sequência até chegar no passo esperado.

Dessa forma, dado o fato que as funções de atualização da interface gráfica também serão executadas, a aplicação vai requerer uma quantidade maior de poder computacional e vai parar de responder até que a execução dos passos seja finalizada.

Pela razão citada acima foi necessário criar uma segunda forma de implementação de controle de fluxo. Nessa implementação, o estado das estruturas em cada passo é salvo em uma \textit{array} usando apenas as estruturas de dados sem modificar os elementos da interface gráfica. Dessa forma, a aplicação consegue pré-calcular os passos da execução sem muito esforço. Assim, quando as ações de controle de fluxo são executadas, os estados salvos são carregados e a interface gráfica é atualizada de acordo com eles.

\section{Implementação das animações dos algoritmos}
As mudanças de estados que ocorrem nos algoritmos podem ser melhor compreendidas se poderem ser visualizadas como transições ao invés de mostrar as mudanças saltando do estado inicial para o estado final. Usar animações torna a visualização das mudanças muito mais dinâmica.

As primeiras animações implementadas são as transições de estilo. Essas foram implementadas usando a propriedade \textit{CSS transition} que aplica uma transição suave entre mudanças de estilo dos elementos como tamanho e cor. As animações seguintes foram implementadas usando \textit{JavaScript} para serem atualizadas dinamicamente a medida que os elementos fossem modificados, mudando de tamanho ou posição na interface gráfica.

Começando com a animação de destaque dos dados dentro de um elemento da interface. Essa animação é usada quando os dados de um elemento são atualizados. Seja quando um item é adicionado a um elemento como inserções em \textit{arrays} ou quando um item é usado como chave para acessar o item de outro elemento como um acesso a um elemento de um \textit{map}. Essa animação põe um item da interface gráfica em destaque com um bloco colorido exibido atrás dele. Quando itens são adicionados às representações gráficas das tabelas de \textit{parse}, as linhas e colunas também são destacadas com uma cor de fundo.

Várias estruturas de dados são representadas na interface gráfica por elementos com itens dispostos na vertical ou horizontal. Iterações com \textit{for loops} feitas sobre esses itens também são representadas na interface gráfica. Para demonstrar cada iteração, uma moldura se movimenta sobre os elementos deixando claro a qual item cada iteração se refere.

Alguns dados são movidos entre elementos como no algoritmo \textit{first} que move símbolos da direita de uma produção para o conjunto \textit{first}. Para demonstrar essa transferência entre os elementos na interface gráfica, uma linha é exibida, estendendo-se do item de origem até o elemento de destino com uma animação de \textit{morphing}. Essa animação é feita usando a biblioteca \textit{anime.js}\footnote{https://animejs.com} que calcula posições intermediárias entre os pontos de origem e destino de um \textit{SVG Path} conforme a duração da animação.

\section{Avaliação da ferramenta}
\label{sec:avaliacao}

A avaliação da ferramenta VANSI foi estruturada para investigar tanto sua usabilidade quanto sua eficácia pedagógica. O estudo adotou uma abordagem metodológica mista, combinando coleta automatizada de dados de interação com questionários estruturados de \textit{feedback}. O estudo envolveu 23 estudantes voluntários que haviam cursado a disciplina de Compiladores do curso de Ciência da Computação do Campus Quixadá da Universidade Federal do Ceará.

\subsection{Coleta de dados}

O segundo passo da coleta de dados consistiu numa atividade prática aplicada com os alunos voluntários. A atividade era composta por sete tarefas, elaboradas para exercitar as principais funcionalidades da ferramenta e avaliar sua completude como recurso pedagógico. A sequência de tarefas estruturou-se da seguinte forma:

\begin{enumerate}[label=\arabic*.]
    \item Geração de gramática a partir do conjunto \textit{First}: Os discentes receberam um conjunto \textit{First} e devem fornecer uma gramática que gere esse conjunto, correspondendo à questão 1 da atividade.

    \item Geração de gramática a partir do conjunto \textit{Follow}: A partir de um conjunto \textit{Follow} fornecido, os participantes devem construir uma gramática que gere esse conjunto, correspondendo à questão 2.

    \item Geração de gramática a partir da tabela \textit{SLR}: Dada uma tabela \textit{SLR}, os discentes devem fornecer uma gramática que gere essa tabela, correspondendo à questão 3.

    \item Geração de gramática a partir da tabela \textit{LR}(1): A partir de uma tabela \textit{LR}(1) fornecida, os participantes devem construir uma gramática que gere essa tabela, correspondendo à questão 4.

    \item Cálculo dos itens canônicos de uma gramática LR(0): Dada uma gramática, os discentes devem fornecer o conjunto de itens canônicos LR(0) da gramática, correspondendo à questão 5.

    \item Análise sintática \textit{LL}(1): Os participantes devem mostrar os passos da análise sintática \textit{LL}(1) da cadeia ``a b a c'' para uma gramática fornecida, correspondendo à questão 6.

    \item Construção de gramática \textit{LR} para entradas específicas: Finalizando com a construção de uma gramática \textit{LR} que aceita as entradas especificadas, correspondendo à questão 7.
\end{enumerate}


Complementando os dados objetivos de interação, o terceiro passo de coleta consistiu em um questionário estruturado de \textit{feedback} aplicado imediatamente após a conclusão da atividade prática. O questionário, composto por 21 questões, foi organizado em quatro seções principais, conforme detalhado no Quadro \ref{qua:questionario-estrutura}. A ativide combinou escalas \textit{Likert} de cinco pontos para questões quantitativas e alternativas fechadas para questões categóricas, reservando uma questão aberta final para comentários e sugestões livres.

A seção de informações gerais buscou caracterizar o perfil dos participantes em relação à experiência prévia e nível de conhecimento. A seção de interface e usabilidade avaliou aspectos como facilidade de navegação, clareza dos elementos e organização do \textit{layout}. A seção de recursos pedagógicos trata da utilidade percebida das diferentes visualizações, animações e mecanismos de controle. Por fim, a avaliação geral coletou impressões subjetivas sobre a ferramenta e intenções de uso futuro.

\subsection{Análise dos dados}

A análise dos dados coletados adotou uma abordagem de métodos mistos, integrando técnicas quantitativas e qualitativas para uma compreensão multidimensional dos resultados. A análise quantitativa concentrou-se inicialmente na estatística descritiva das métricas de uso extraídas dos \textit{logs}, calculando médias, medianas e distribuições de frequência para os diferentes tipos de interação.

Além disso, técnicas de visualização foram empregadas com a geração de \textit{heatmaps} de distribuição de cliques e linhas do tempo de interação, facilitando a identificação de padrões comportamentais. A identificação desses padrões foi feita manualmente através da comparação entre as vizualizações geradas e o \textit{feedback} dos participantes.

Os \textit{heatmaps} de cliques foram gerados utilizando as coordenadas de clique registradas nos \textit{logs}, aplicando um mapa de calor sobre uma imagem interface da ferramenta para destacar as áreas de maior interação. As imagens da interface presentes nos \textit{heatmaps} foram geradas automaticamente por um \textit{script Common JavaScript} executado no \textit{Node.js}\footnote{https://nodejs.org}. O \textit{script} utiliza a biblioteca \textit{puppeteer} para lançar uma instância de um navegador, acessar o \textit{url} da ferramenta, navegar até a tela desejada e fazer uma captura de tela tudo de forma automatizada. O \textit{script} está disponível publicamente no repositório do projeto\footnote{https://github.com/fagner02/parsers-svelte.git}.

Os rótulos de tempo (\textit{timestamps}) dos cliques nas abas foram usados para determinar qual aba estava ativa durante cada evento registrado. Isso permitiu a geração de \textit{screenshots} específicas de cada aba da aplicação. Os rótulos de tempo também permitiram o cálculo de médias de tempo gasto em cada aba da aplicação.

As linhas do tempo de interação foram construídas plotando os eventos registrados ao longo do tempo, permitindo a visualização de sequências e durações de uso dos diferentes componentes da interface gráfica. Os gráficos de linha de tempo também marcam as transições entre as abas que são identificadas por cliques em elementos de navegação registrados nos \textit{logs}.

Por fim, a análise qualitativa focou nos comentários livres do questionário. Essa análise buscou identificar temas recorrentes, sugestões de melhoria e percepções subjetivas que complementassem os dados quantitativos. A análise feita a partir de multiplas fontes, integrando dados de \textit{logs}, respostas escalares e comentários textuais, permitiu uma validação cruzada dos achados e uma compreensão mais rica da experiência dos usuários.

\subsection{Ferramentas e considerações técnicas}

Para o processamento e análise dos dados, foram desenvolvidos \textit{scripts} em \textit{Python} que utilizam as bibliotecas \textit{pandas} e \textit{numpy} para manipulação e análise estatística dos dados tabulares, \textit{matplotlib} e \textit{seaborn} para geração de visualizações e gráficos. Todo o código desenvolvido para a análise está disponível publicamente no repositório do projeto, garantindo a transparência e reprodutibilidade dos procedimentos analíticos.

A condução do estudo também considerou aspectos éticos da pesquisa envolvendo seres humanos. Foi obtido o consentimento informado de todos os participantes, garantindo o anonimato completo dos dados pessoais e a utilização exclusiva das informações para fins acadêmicos e de pesquisa.

Esta metodologia de avaliação, ao integrar dados objetivos de interação com percepções subjetivas dos usuários, permitirá não apenas quantificar a aceitação da ferramenta, mas também compreender os mecanismos pelos quais a visualização interativa pode facilitar o aprendizado de algoritmos complexos de análise sintática.
